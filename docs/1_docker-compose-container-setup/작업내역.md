## [2026-05-16] Docker Compose 컨테이너 환경 구성

### 작업 내용
- 프로젝트 디렉토리 구조 생성 (`proto/`, `server/`, `client/`, `shared/input/`, `shared/output/`)
- `proto/file_streamer.proto` 작성 — Bidirectional Streaming RPC 서비스 및 메시지 정의
- `server/`, `client/` 각각 `requirements.txt`, `Dockerfile`, 스켈레톤 `main.py` 작성
- `docker-compose.yml` 작성 — 브리지 네트워크, 볼륨 마운트, depends_on 설정
- `docker compose up --build` 실행 후 클라이언트 연결 성공 확인

### 주요 결정 사항
- **UploadRequest / UploadResponse에 `oneof` 사용**: 같은 스트림 안에서 메타데이터 패킷과 청크 패킷(또는 진행률과 최종 결과)을 하나의 메시지 타입으로 구분하기 위해 proto3 `oneof` 적용
- **grpcio / grpcio-tools 버전 고정 (`1.73.0`)**: 두 패키지의 내부 인터페이스가 버전마다 달라지므로 반드시 동일 버전으로 고정해야 stub 코드와 런타임 불일치 오류를 방지할 수 있음. 빌드 재현성도 확보
- **Dockerfile 빌드 컨텍스트를 프로젝트 루트로 설정**: `COPY ../proto/...`는 Docker 빌드 컨텍스트 밖을 참조해 오류가 발생하므로, `docker-compose.yml`에서 `context: .`로 루트를 빌드 컨텍스트로 지정하고 Dockerfile 내 경로를 루트 기준으로 작성
- **shared 디렉토리를 `input/` / `output/`으로 분리**: 원본 파일과 전송 파일을 명확히 분리해 무결성 테스트 시 비교를 용이하게 함. 클라이언트는 `input`을 read-only(`:ro`)로 마운트해 원본 파일 보호
- **호스트 디렉토리 마운트 방식 채택**: Docker named volume 대신 호스트 마운트를 사용해 원본 파일을 호스트에서 직접 배치하고 결과 파일을 바로 확인할 수 있도록 함

### 이슈 & 해결
- **`COPY ../proto/...` 빌드 오류**: Dockerfile에서 빌드 컨텍스트 밖 경로를 참조하면 오류 발생. `docker-compose.yml`의 `context: .`로 프로젝트 루트를 빌드 컨텍스트로 설정하고, Dockerfile 내 COPY 경로를 루트 기준으로 수정하여 해결
